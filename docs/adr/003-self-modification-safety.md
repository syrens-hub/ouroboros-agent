# ADR 003: 自修改安全模型（4 层防护）

## 上下文（Context）

自修改（self-modification）是 Ouroboros Agent 的核心能力，也是其最大风险来源。系统能够：

- 读取和写入自身源代码
- 运行测试验证变更
- 执行 `git commit` 将变更持久化

这种能力如果失控，可能导致：
1. **核心系统被破坏**：AI 误删或篡改 `core/` 层代码，导致系统无法启动。
2. **供应链投毒**：AI 引入未经审核的外部网络依赖。
3. **不可逆数据丢失**：删除数据库迁移文件或配置文件。
4. **无限递归修改**：AI 进入自我修改的无限循环。

因此，必须在"能力开放"与"风险可控"之间建立严格的多层防线。

---

## 决策（Decision）

我们决定实施 **4 层自修改安全模型**，任何代码变更必须通过全部 4 层检查才能最终落盘：

### 第 1 层：宪法守卫（Constitution Guard）

硬编码的不可逾越规则，位于 `core/safety/constitution.ts`：

- **禁止删除 core/ 目录下任何文件**
- **禁止修改标记为 immutable 的文件**（由 `.ouroboros/immutable.lock` 定义）
- **禁止引入新的网络依赖**（`package.json` 中新增 `dependencies` 需人工审批）
- **禁止修改宪法文件本身**
- **禁止删除或篡改 git 历史**

宪法规则**无运行时覆盖开关**，任何违反直接拒绝并记录审计日志。

### 第 2 层：语法验证（Syntax Validation）

变更落盘前，在临时副本上执行：

```bash
tsc --noEmit
```

- TypeScript 编译失败 → 拒绝变更，返回错误详情给 AI
- 编译通过 → 进入第 3 层

### 第 3 层：原子写入（Atomic Write）

文件系统操作采用 **临时文件 → 验证 → 重命名** 的两阶段提交：

1. 写入 `.ouroboros/tmp/mutation-<uuid>/...`
2. 在该临时副本上运行语法验证和基础 lint
3. 全部通过后，执行原子 `fs.renameSync()` 覆盖目标文件
4. 任意步骤失败 → 清理临时目录，原文件不受影响

### 第 4 层：金丝雀测试（Canary Tests）

变更落盘后，自动触发受影响的测试套件：

1. 识别变更涉及的 skill / core 模块（通过 import 依赖分析）
2. 运行相关 Vitest 测试（`npx vitest run --related <files>`）
3. **全部通过**：变更正式生效，允许后续 git commit
4. **任何失败**：自动执行 **Rollback**（从 git HEAD 恢复），并将失败报告回传给 AI

---

## 权衡（Trade-offs）

### 多层模型的优势

- **纵深防御**：单点失效不会导致灾难；宪法守卫作为最终保底。
- **可观测性**：每层都有独立日志，便于事后审计和问题定位。
- **渐进信任**：新 Agent / 新场景默认走完全部流程；高置信度场景未来可考虑缓存。

### 多层模型的劣势

- **延迟增加**：完整 4 层流程在大型代码库上可能耗时 10–30 秒。
- **资源消耗**：频繁的自修改会触发大量 TypeScript 编译和测试运行。

---

## 后果（Consequences）

### 积极后果

1. **核心不可破坏**：即使 AI 产生有害意图或幻觉，宪法守卫阻止对系统根基的修改。
2. **变更可回滚**：原子写入 + 金丝雀测试失败自动回滚，保证文件系统始终处于可运行状态。
3. **质量内建**：语法验证迫使 AI 生成类型安全的代码，减少运行时错误。
4. **审计追踪**：每一层决策都写入 `evolution_audit_log` 表，支持事后溯源。

### 消极后果与缓解措施

| 风险 | 缓解措施 |
|---|---|
| 完整安全流程慢，影响迭代速度 | 引入增量编译；测试仅运行与变更相关的子集；提供 `--fast-mutation` 模式（跳过第 4 层，仅供开发调试） |
| 宪法规则过于严格，限制合理演进 | 宪法文件仅包含"生存底线"规则；常规重构不受限制；每季度由维护者评审宪法内容 |
| 金丝雀测试 flaky 导致误回滚 | 设置重试机制（最多 3 次）；区分"确定性失败"与"超时/ flaky" |
| 需要紧急绕过所有安全层（灾难恢复） | 预留 `OUROBOROS_EMERGENCY_OVERRIDE=1` 环境变量，启用时记录最高级别审计日志并触发即时告警 |

### 中立后果

- 自修改 skill 的实现复杂度显著提高（需与 4 层系统交互）。
- 用户需要理解"为什么我的修改被拒绝了"——需要建设错误解释和引导机制。

---

## 替代方案（Alternatives Considered）

### 替代 A：完全信任模型（无安全层）

- **拒绝原因**：与项目可靠性目标完全冲突；AI 幻觉或提示注入可能导致自我毁灭。

### 替代 B：外部沙箱（Docker / 隔离进程）

- **拒绝原因**：增加了运行时依赖和构建复杂度；自修改系统需要直接访问宿主文件系统以实现 git 提交和 IDE 集成，沙箱化会削弱核心价值。

### 替代 C：代码审查代理（AI 审查 AI）

- **拒绝原因**：作为补充机制可行，但不能替代确定性规则；AI 审查同样存在被绕过的风险。未来可在第 2–3 层之间加入可选的审查代理。
